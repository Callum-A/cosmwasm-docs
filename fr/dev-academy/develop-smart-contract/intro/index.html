Anatomie d'un contrat intelligent

Un contrat intelligent peut être considéré comme une instance d'un objet singleton dont l'état interne est conservé sur la blockchain . Les utilisateurs peuvent déclencher des changements d'état en lui envoyant des messages JSON, et les utilisateurs peuvent également interroger son état en envoyant à une requête formatée comme un message JSON.

En tant que rédacteur de contrat intelligent, votre travail consiste à définir 3 fonctions qui définissent l'interface de votre contrat intelligent :

  • instantiate(): un constructeur qui est appelé pendant l'instanciation du contrat pour fournir l'état initial
  • execute(): est appelé lorsqu'un utilisateur veut invoquer une méthode sur le smart contract
  • query(): est appelé lorsqu'un utilisateur veut obtenir des données d'un contrat intelligent.

Dans cette section, nous allons définir nos messages attendus et leur mise en œuvre.

Commencez par un modèle​

Dans votre répertoire de travail, vous voudrez utiliser cargo-generate pour démarrer votre contrat intelligent avec la structure du dossier et les options de construction recommandées :

# install cargo-generate
cargo install cargo-generate --features vendored-openssl
cargo generate --git https://github.com/CosmWasm/cosmwasm-template.git --name my-first-contract
cd my-first-contract

Cela vous aidera à démarrer en fournissant le modèle de base et la structure d'un contrat intelligent. Vous trouverez dans le fichier src/lib.rs que les points d'entrée standard de CosmWasm instantiate(), execute(), et query() sont correctement exposés et connectés.

État du contrat​

Le modèle de départ a l'état de base suivant :

  • une structure singleton État contenant :
    • un nombre entier de 32 bits ``
    • une adresse propriétaire
// src/state.rs
use schemars::JsonSchema ;
use serde: :{Deserialize, Serialize};

use cosmwasm_std: :{CanonicalAddr, Storage};
use cosmwasm_storage::{singleton, singleton_read, ReadonlySingleton, Singleton} ;

#[derive(Serialize, Deserialize, Clone, Debug, PartialEq, JsonSchema)]
pub struct State {
  pub count : i32,
  pub owner : Addr,
}

Les contrats intelligents ont la possibilité de conserver un état persistant grâce au système natif LevelDB, un magasin clé-valeur basé sur des octets . Ainsi, toute donnée que vous souhaitez conserver doit se voir attribuer une clé unique à partir de laquelle les données peuvent être indexées et récupérées ultérieurement à l'adresse .

// src/state.rs
pub const STATE : Item<State> = Item::new("state") ;

Dans l'exemple ci-dessus, la clé "state" est utilisée comme préfixe.

Les données ne peuvent être conservées que sous forme d'octets bruts, de sorte que toute notion de structure ou de type de données doit être exprimée sous la forme d'une paire de fonctions de sérialisation et de désérialisation . Par exemple, les objets doivent être stockés sous forme d'octets. Vous devez donc fournir à la fois la fonction qui code l'objet en octets pour l'enregistrer sur la blockchain, et la fonction qui décode les octets en types de données que votre logique de contrat peut comprendre. Le choix de la représentation des octets vous appartient, pour autant que fournisse une correspondance propre et bidirectionnelle.

Heureusement, l'équipe CosmWasm a fourni des crates utilitaires tels que cosmwasm_storage, qui fournit des abstractions de haut niveau pratiques pour les conteneurs de données tels qu'un "singleton" et un "bucket", qui fournissent automatiquement la sérialisation et la désérialisation pour les types couramment utilisés tels que les structs et les nombres Rust.

Remarquez comment la structure State contient à la fois count et owner. En outre, l'attribut derive est appliqué pour auto-implémenter certains traits utiles :

  • Serialize: fournit la sérialisation
  • Deserialize: fournit une désérialisation
  • Clone: rend notre structure copiable
  • Debug: permet d'imprimer notre structure sur une chaîne de caractères.
  • PartialEq: nous donne une comparaison d'égalité
  • JsonSchema: génère automatiquement un schéma JSON pour nous

Addr qui représente une adresse Bech32 lisible par l'homme, préfixée par wasm....

InstantiateMsg​

Le InstantiateMsg est fourni lorsqu'un utilisateur crée un contrat sur la blockchain par le biais d'un MsgInstantiateContract. Ce site fournit au contrat sa configuration ainsi que son état initial.

Sur le CosmWasm, le téléchargement du code d'un contrat et l'instanciation d'un contrat sont considérés comme des événements distincts , contrairement à Ethereum. This is to allow a small set of vetted contract archetypes to exist as multiple instances sharing the same base code but configured with different parameters (imagine one canonical ERC20, and multiple tokens that use its code).

Exemple​

Pour notre contrat, nous attendons du créateur du contrat qu'il fournisse l'état initial dans un message JSON :

{
  "count" : 100
}

Définition du message​

// src/msg.rs

use schemars::JsonSchema;
use serde::{Deserialize, Serialize};

#[derive(Serialize, Deserialize, Clone, Debug, PartialEq, JsonSchema)]
pub struct InstantiateMsg {
  pub count: i32,
}

Logique​

Nous définissons ici notre premier point d'entrée, la fonction instantiate(), où le contrat est instancié et reçoit son message InstantiateMsg. Nous extrayons le compte du message et établissons notre état initial, où :

  • compte se voit attribuer le compte du message
  • propriétaire est attribué à l'expéditeur du MsgInstantiateContract ``
// src/contract.rs
#[cfg_attr(not(feature = "library"), entry_point)]
pub fn instantiate(
  deps : DepsMut,
  _env : Env,
  info : MessageInfo,
  msg : InstantiateMsg,
) -> Result<Response, ContractError> {
  let state = State {
    count : msg.count,
    owner : info.sender.clone(),
  } ;
  set_contract_version(deps.storage, CONTRACT_NAME, CONTRACT_VERSION)? ;
  STATE.save(deps.storage, &state)? ;

  Ok(Response::new()
    .add_attribute("method", "instantiate")
    .add_attribute("owner", info.sender)
    .add_attribute("count", msg.count.to_string()))
}

ExecuteMsg​

Le fichier ExecuteMsg est un message JSON transmis à la fonction execute() par l'intermédiaire d'un contrat MsgExecuteContract. Contrairement au message InstantiateMsg, le message ExecuteMsg peut exister sous plusieurs types de messages différents, afin de tenir compte des différents types de fonctions qu'un contrat intelligent peut exposer à un utilisateur. La fonction execute() démultiplexe ces différents types de messages vers sa logique de traitement des messages appropriée.

Exemple​

Increment​

Tout utilisateur peut incrémenter le compte actuel par 1.

{
  "increment": {}
}

Reset​

Seul le propriétaire peut remettre le compte à un nombre spécifique.

{
  "reset" : {
    "count" : 5
  }
}

Définition du message​

Comme pour notre ExecuteMsg, nous utiliserons un enum `pour multiplexer les différents types de messages que notre contrat peut comprendre . L'attributserderéécrit nos clés d'attribut en majuscules et en minuscules, de sorte que nous auronsincrementetresetau lieu deincrementetreset` lors de la sérialisation et de la désérialisation à travers JSON.

// src/msg.rs
#[derive(Serialize, Deserialize, Clone, Debug, PartialEq, JsonSchema)]
#[serde(rename_all = "snake_case")]
pub enum ExecuteMsg {
  Increment {},
  Reset { count: i32 },
}

Logique​

// src/contract.rs
#[cfg_attr(not(feature = "library"), entry_point)]
pub fn execute(
  deps : DepsMut,
  _env : Env,
  info : MessageInfo,
  msg : ExecuteMsg,
) -> Result<Response, ContractError> {
  match msg {
    ExecuteMsg::Increment {} => try_increment(deps),
    ExecuteMsg::Reset { count } => try_reset(deps, info, count),
  }
}

Il s'agit de notre méthode execute() , qui utilise la correspondance de motifs de Rust pour acheminer le message reçu ExecuteMsg vers la logique de traitement appropriée , en envoyant un appel try_increment() ou try_reset() en fonction du message reçu.

pub fn try_increment(deps : DepsMut) -> Result<Response, ContractError> {
  STATE.update(deps.storage, |mut state| -> Result<_, ContractError> {
    state.count = 1 ;
    Ok(state)
  })? ;

  Ok(Response::new().add_attribute("method", "try_increment"))
}

Il est assez simple de suivre la logique de try_increment(). Nous acquérons une référence mutable au stockage pour mettre à jour l'élément situé à la clé "state", rendue accessible par la fonction de commodité STATE définie dans le fichier src/state.rs. Nous mettons ensuite à jour le compte de l'état actuel en renvoyant un résultat Ok avec le nouvel état. Enfin, nous terminons l'exécution du contrat avec un accusé de réception en renvoyant un résultat Ok avec la réponse par défaut ``.

Dans cet exemple, la réponse par défaut Response est utilisée pour des raisons de simplicité. Toutefois, la réponse `` peut être créée manuellement pour fournir les informations suivantes :

  • messages: une liste de messages. C'est là que les contrats intelligents exécutent d'autres contrats intelligents ou utilisent des modules natifs.
  • attributs: une liste de paires clé-valeur pour définir les événements SDK émis qui peuvent être souscrits par les clients et analysés par le bloc.
  • événements: événements supplémentaires, personnalisés et distincts de l'événement principal wasm . Le type sera précédé de wasm- . Explorateurs et applications pour signaler les événements importants ou les changements d'état survenus pendant l'exécution.
  • données: données supplémentaires que le contrat renvoie au client.
// src/contract.rs
pub fn try_reset(deps : DepsMut, info : MessageInfo, count : i32) -> Result<Response, ContractError> {
  STATE.update(deps.storage, |mut state| -> Result<_, ContractError> {
    if info.sender != state.owner {
      return Err(ContractError::Unauthorized {}) ;
    } }
    state.count = count ;
    Ok(state)
  })? ;
  Ok(Response::new().add_attribute("method", "reset"))
}

La logique de la réinitialisation est très similaire à celle de l'incrémentation, sauf que cette fois, nous vérifions d'abord que l'expéditeur du message est autorisé à invoquer la fonction de réinitialisation.

QueryMsg​

Exemple​

Le contrat type ne prend en charge qu'un seul type de QueryMsg:

Balance​

La demande :

{
  "get_count": {}
}

Qui devrait revenir :

{
  "count" : 5
}

Définition du message​

Pour prendre en charge les requêtes sur notre contrat de données, nous devons définir à la fois un format QueryMsg (qui représente les requêtes ), et fournir la structure du résultat de la requête -- CountResponse dans ce cas. Nous devons le faire parce que query() va renvoyer des informations à l'utilisateur par le biais de JSON dans une structure et nous devons faire connaître la forme de notre réponse .

Ajoutez ce qui suit à votre src/msg.rs:

// src/msg.rs
#[derive(Serialize, Deserialize, Clone, Debug, PartialEq, JsonSchema)]
#[serde(rename_all = "snake_case")]
pub enum QueryMsg {
  // GetCount renvoie le compte actuel sous la forme d'un nombre encodé en json
  GetCount {},
}

// Nous définissons une structure personnalisée pour chaque réponse à une requête
#[derive(Serialize, Deserialize, Clone, Debug, PartialEq, JsonSchema)]
pub struct CountResponse {
  pub count : i32,
}

Logique​

La logique de query() devrait être similaire à celle de execute(), sauf que, puisque query() est appelé sans que l'utilisateur final n'effectue de transaction, nous omettons l'argument env car il n'y a pas d'information.

// src/contract.rs
#[cfg_attr(not(feature = "library"), entry_point)]
pub fn query(deps : Deps, _env : Env, msg : QueryMsg) -> StdResult<Binary> {
  match msg {
    QueryMsg::GetCount {} => to_binary(&query_count(deps) ?),
  }
}

fn query_count(deps : Deps) -> StdResult<CountResponse> {
  let state = STATE.load(deps.storage)? ;
  Ok(CountResponse { count: state.count })
}

Élaboration du contrat​

Pour construire votre contrat, exécutez la commande suivante. Cela permettra de vérifier l'existence d'éventuelles erreurs préliminaires avant d'optimiser.

cargo wasm

Optimiser votre construction​

NOTE Vous devez avoir installé Docker pour exécuter cette commande. :::

Vous devrez vous assurer que le binaire WASM de sortie est aussi petit que possible afin de minimiser les frais et de rester sous la limite de taille de pour la blockchain. Exécutez la commande suivante dans le répertoire racine du dossier du projet de votre contrat intelligent Rust .

docker run --rm -v "$(pwd)":/code \
  --mount type=volume,source="$(basename "$(pwd)")_cache",target=/code/target \
  --mount type=volume,source=registry_cache,target=/usr/local/cargo/registry \
  cosmwasm/rust-optimizer:0.12.0

Il en résultera une construction optimisée de artifacts/my_first_contract.wasm dans votre répertoire de travail.

(Facultatif) Ajoutez la commande ci-dessus dans Cargo.toml pour un accès rapide.

Cela permet d'exécuter des commandes de script personnalisées de manière similaire à package.json dans l'écosystème Node.

Installer cargo-run-script

cargo install cargo-run-script

Ajouter le script dans Cargo.toml

[package.metadata.scripts]
optimize = "" "docker run --rm -v "$(pwd)":/code \
  --mount type=volume,source="$(basename "$(pwd)")_cache",target=/code/target \
  --mount type=volume,source=registry_cache,target=/usr/local/cargo/registry \
  cosmwasm/rust-optimizer:0.12.0
"""

Exécutez la commande :

cargo run-script optimize

Schémas​

Afin d'utiliser la génération automatique de schémas JSON, nous devons enregistrer chacune des structures de données pour lesquelles nous avons besoin de schémas .

// examples/schema.rs

use std ::env::current_dir ;
use std ::fs::create_dir_all ;

use cosmwasm_schema: :{export_schema, remove_schemas, schema_for};

use my_first_contract ::msg::{CountResponse, ExecuteMsg, InstantiateMsg, QueryMsg} ;
use my_first_contract ::state::State ;

fn main() {
  let mut out_dir = current_dir().unwrap() ;
  out_dir.push("schema") ;
  create_dir_all(&out_dir).unwrap() ;
  remove_schemas(&out_dir).unwrap() ;

  export_schema(&schema_for !(InstantiateMsg), &out_dir) ;
  export_schema(&schema_for !(ExecuteMsg), &out_dir) ;
  export_schema(&schema_for !(QueryMsg), &out_dir) ;
  export_schema(&schema_for !(State), &out_dir) ;
  export_schema(&schema_for !(CountResponse), &out_dir) ;
}

Vous pouvez ensuite construire les schémas avec :

cargo schema

Vos schémas nouvellement générés devraient être visibles dans votre répertoire schema/ . Voici un exemple de schema/query_msg.json.

{
  "$schema" : "http://json-schema.org/draft-07/schema#",
  " title " : "QueryMsg",
  "anyOf" : [
    {
      "type" : "object",
      "required" : [
        "get_count"
      ],
      "properties" : {
        "get_count" : {
          "type" : "object"
        }
      },
      "additionalProperties" : false
    }
  ]
}

Vous pouvez utiliser un outil en ligne tel que JSON Schema Validator pour tester votre entrée par rapport au schéma JSON généré.
